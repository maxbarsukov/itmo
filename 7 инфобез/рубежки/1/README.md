#### 6. Задание

Компания разрабатывает мобильное приложение для заказа такси. В приложении обрабатываются геолокация пользователей, история поездок и платежные данные.

**Вопрос:** Проведите качественную оценку рисков по методике DREAD для угрозы «Утечка платежных данных через уязвимость в API».
Предложите 2 технические и 1 организационную меру защиты.

Критерии оценки:
- Корректность оценки по DREAD (1 балл)
- Релевантность технических мер (1 балл)
- Обоснованность организационной меры (1 балл)

**Ответ:**

1. Damage 3 - утечка платежных данных ведёт к финансовым потерям, штрафам (ФЗ-152), репутационному ущербу;
2. Reproducibility 3 - уязвимость в API позволяет многократно и стабильно извлекать данные;
3. Exploitability 2 - требуется базовые навыки, но возможна автоматизация;
4. Affected users 3 - все пользователи с привязанными картами;
5. Discoverability 2 - уязвимость может быть найдена сканерами или пентестом при целенаправленном тестировании API.

Итоговый балл DREAD (3+3+2+3+2)/5 = 2.6, значит высокий риск.

Технические меры:
- Web Application Firewall для фильтрации вредоносных запросов к API;
- Статический и динамический анализ кода (SAST/DAST) API, интегрированный в CI/CD для поиска уязвимостей до попадания в продакшен.

Организационная мера:
- Внедрение и соблюдение политики безопасной разработки (Secure SDLC), включая обязательное проведение моделей угроз для новых функций API и код-ревью с фокусом на безопасность.

#### 7. Задание
Крупный банк планирует внедрить систему управления инцидентами ИБ.

**Вопрос:** Разработайте чек-лист первоочередных действий при обнаружении фишинговой атаки на сотрудников.
Опишите 3 ключевых метрики эффективности организационных мер защиты, которые банк должен отслеживать.
Объясните, как политика чистого стола снижает риски утечки информации.

Критерии оценки:
- Практическая применимость чек-листа (1 балла)
- Релевантность метрик эффективности (1 балла)

**Ответ:**

Чек-лист:
1. Заблокировать подозрительные письма/ссылки на почтовом шлюзе;
2. Уведомить всех сотрудников о фишинговой атаке через внутренние каналы;
3. Собрать и сохранить доказательства (заголовки писем, URL, скриншоты) для анализа;
4. Оценить масштаб;
5. Выявить круг сотрудников, которые могли взаимодействовать с письмом (открыли, перешли по ссылке, ввели данные);
6. Для затронутых сотрудников принудительная смена паролей, проверка рабочих станций на наличие признаков компрометации, анализ логов их активностей;
7. Обновить правила фильтрации в почтовой системе.

Метрики:
1. Среднее время обнаружения инцидента - насколько быстро система мониторинга и сотрудники выявляют угрозы;
2. Среднее время реагирования на инцидент - эффективность процессов и команды реагирования;
3. Процент сотрудников, успешно прошедших тестовые фишинговые кампании - эффективность программ обучения и осведомленности.

Политика чистого стола требует, чтобы сотрудники по окончании рабочего дня убирали все конфиденциальные документы в запираемые шкафы, а также блокировали рабочие станции, что предотвращает несанкционированный физический доступ к информации, снижает риски утечки данных в условиях внутренних угроз или социальной инженерии, а также поддерживает принцип минимальных привилегий.
