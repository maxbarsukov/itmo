> В тесте нет никаких случайных опечаток. Варианты ответов типа «парольной НЕсложности», «ПЕРЕтекстинга» и «БЕЗвоздействия» — это намеренные ловушки, и нужно понимать всё буквально.

- [ ] Элементы действий разработчика;
- [ ] Элементы содержания и представления документированных свидетельств;
- [ ] Политика парольной несложности;
- [ ] Элементы содержания документированных свидетельств;
- [ ] Настройка межсетевого экрана;
- [ ] Элементы действий архитектора программного обеспечения;
- [ ] Шифрование данных;
- [ ] Элементы действий оценщика.

- [ ] Общие персонифицированные данные;

- [ ] Неправильная настройка сервера;

- [ ] Слабые алгоритмы шифрования;
- [ ] Уязвимости компонентов с известными уязвимостями;
- [ ] Небезопасные десериализации.

#### 6. Задание

Компания разрабатывает мобильное приложение для заказа такси. В приложении обрабатываются геолокация пользователей, история поездок и платежные данные.

**Вопрос:** Проведите качественную оценку рисков по методике DREAD для угрозы «Утечка платежных данных через уязвимость в API».
Предложите 2 технические и 1 организационную меру защиты.

Критерии оценки:
- Корректность оценки по DREAD (1 балл)
- Релевантность технических мер (1 балл)
- Обоснованность организационной меры (1 балл)

#### 7. Задание
Крупный банк планирует внедрить систему управления инцидентами ИБ.

**Вопрос:** Разработайте чек-лист первоочередных действий при обнаружении фишинговой атаки на сотрудников.
Опишите 3 ключевых метрики эффективности организационных мер защиты, которые банк должен отслеживать.
Объясните, как политика чистого стола снижает риски утечки информации.

Критерии оценки:
- Практическая применимость чек-листа (1 балла)
- Релевантность метрик эффективности (1 балла)
